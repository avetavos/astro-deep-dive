## The idea in one sentence

**Astro is static by default, but you opt into the server whenever a page needs request-time work** — the logged-in user, a form submission, live data, an auth check. The server is a tool you reach for deliberately, not the default substrate.

Everything in this module is about that opt-in: rendering per request, handling requests directly, calling server logic type-safely, and running code before every request.

## Four ways to run server code

<Mermaid title="The server-side surface of Astro" code={overviewDiagram} />

export const overviewDiagram = `flowchart TB
  req["Incoming request"] --> mw["Middleware onRequest\n(runs first, every request)"]
  mw --> page["On-demand page\n(prerender false)"]
  mw --> api["API endpoint\n(GET/POST handler)"]
  mw --> act["Action handler\n(defineAction)"]
  page --> resp["Response"]
  api --> resp
  act --> resp`;

- **On-demand pages** render HTML per request, so they can read cookies, the user, and query params.
- **API endpoints** are non-HTML routes — return JSON, a file, a webhook reply — as a raw `Response`.
- **Actions** are type-safe server functions you call from the client with validated input, ideal for mutations and forms.
- **Middleware** runs before every request, the place for auth, logging, and shared per-request data.

<Callout title="The one-line takeaway">
Server-side Astro is **opt-in**: static by default, you reach for the server only when a page needs request-time data or logic. Four tools cover it — **on-demand pages** (HTML per request), **API endpoints** (raw `Response` for non-HTML), **Actions** (type-safe validated server functions for the client), and **middleware** (code that runs before every request, with sessions for per-user state). All of them require an adapter.
</Callout>
